# Jira Leaderboard

A real-time leaderboard application built using **Atlassian Forge**, **TypeScript**, and **React** as part of the **UNSW COMP9900 Industry Capstone** in collaboration with **Atlassian**.

The application integrates directly into **Jira Spaces** and enables teams to recognise and encourage contributions through a configurable, real-time leaderboard. Administrators can customise scoring rules, filter issues using JQL, and view live rankings that automatically update as work is completed. The application leverages Jira Cloud REST APIs, Forge Storage, and Forge Realtime APIs to provide persistent configuration, real-time synchronisation, and a seamless native Jira experience.

---

## App Summary

This project contains an Atlassian Forge application written in **TypeScript** that displays a configurable leaderboard for contributions within Jira Spaces.

### Features

- Real-time leaderboard displaying user rankings
- Configurable scoring rules for administrators
- JQL-based issue filtering
- Real-time synchronisation using Forge Realtime APIs
- Jira Cloud REST API integration
- Persistent configuration using Forge Storage
- Native user interface built with Forge UI Kit
- Automated testing and continuous integration using GitHub Actions

## Quick start (Running the Application)

Running the application requires a few pre-requisites:

- Forge installation. See [Set up Forge](https://developer.atlassian.com/platform/forge/set-up-forge/)
- An Atlassian site to deploy the application
- A Forge app ID (paste the yml file where indicated by comments). Can be obtained with the forge register command. See [CLI reference](https://developer.atlassian.com/platform/forge/cli-reference/register/)
- Run `npm install` in the project directory (see package.json file for dependency list; the app is composed (exclusively) by Atlassian/Forge tooling + React)
- Replace the id variable at the bottom of the yml file with local forge app id
- Run `forge install` (follow Atlassian CLI prompts)
- Run `forge deploy`

## Contributing to this Project

### Code Practices

- Include comments above each function describing what it does/how it works
- Where technical debt will accrue comment "TODO: ...." and describe the remaining functionality to be implemented (this should only be done where necessary; in general avoid accruing technical debt)
- Run the linter and tests before commits (see .eslintrc); GitHub actions enforces this but best to check locally to avoid unneeded remote branches

### Debugging

- Fortunately, as this product is built within the Atlassian toolset it inherits a rich variety of tools for debugging. The most useful of these by far is `forge tunnel`. This command allows you to re-route requests to Atlassian's servers to your local machine. This has two important properties for this project's development:
    - (1) It means that backend logs will appear in terminal (if you type console.log in any backend process, it log will appear in the terminal used to run `forge tunnel`). This greatly simplifies the process of debugging backend functions;
    - (2) It means that your app will update in real time as you code and save your work. This reduces a huge amount of time for debugging as otherwise you have to run `forge deploy` any time you want to see how code changes manifest in the app.
- If you do not want to use `forge tunnel` you can use the Atlassian [developer console](developer.atlassian.com) > "Logs" to see backend logs. But this will not obviate the need to re-deploy with every local change.
- All external api calls must be wrapped in `try` `catch` blocks with appropriate error handling/messaging so that others can debug outside calls (note: this calls are exclusively to Atlassian)
- If you want something else to help debug an issue, include detailed code comments on your branch. This ensures that the core issues are captured in the shared code.

### Submitting PRs

- **Do not push code directly to master**
- Build features in separate branches
- Expected branch syntax: name / (jira code | feature name)
- Submit PR and ask another team member for review
- GitHub actions will run automated tests

### Notes

- Use the `forge deploy` command when you want to persist code changes.
- Use the `forge install` command when you want to install the app on a new site.
- Once the app is installed on a site, the site picks up the new app changes you deploy without needing to rerun the install command.
